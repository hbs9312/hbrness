# 에러 핸들러·i18n 리소스 생성 (F-Err)

ultrathink

당신은 프론트엔드 개발자입니다.
기술 명세서(TS)의 §4 에러 코드 맵을 프론트엔드 코드 상수, 사용자 메시지 핸들러, 다국어 리소스로 변환합니다.

## 컨텍스트 로드

- **프로젝트 설정**: [frontend.md](../../context/frontend.md) — `error_handling` 섹션 **필수**, `api_client`
- **설계 문서**: `docs/designs/phase-1-error-contract.md` (프로젝트 루트에 있으면 참조)
- **태스크-파일 맵**: `.frontflow/task-file-map.md` (있으면)

## 입력

$ARGUMENTS 에서:
1. **기술 명세서(TS)** → `§4 에러 코드 맵` 섹션 (필수). `specflow:generate-ts` 4-tier 포맷 이후 생성된 TS
2. **UI 명세서(UI)** (선택) → 에러 UI 플로우 힌트. `ui_flow` 컬럼이 TS 에 없거나 UI 에 더 구체적 지시가 있을 때 참조

## 태스크-파일 맵 준수

`.frontflow/task-file-map.md`가 존재하면:
1. `impl_skill: impl-error-handling` 항목만 필터
2. `action: create` → 해당 경로에 새 파일 생성
3. `action: modify` → 기존 파일에 행 append (기존 코드 보존)
4. 맵에 없는 파일은 생성하지 않음

맵이 없으면 `frontend.md.error_handling.*` 경로를 기준으로 독자 판단.

## ★ 실행 위치 — impl-api-integration 직전 ★

이 스킬은 map-tasks 이후, **impl-interactions / impl-api-integration 보다 먼저** 실행되는 것이 이상적입니다.
API 통합 시 에러 분기 코드에서 `handleError()` 를 import 해서 사용하도록 유도합니다.

실행 순서:
```
map-tasks → impl-tokens → impl-atoms → impl-composites → impl-pages
          → impl-error-handling → impl-interactions → impl-api-integration
```

`impl-api-integration` 은 이 스킬이 만든 `handler.ts` 를 **import 해서 사용** 합니다. impl-api-integration 이 직접 switch/case 매핑을 인라인하지 않도록 주의 (중복 금지).

## ★ 계약: TS 가 source of truth — backend 와 code 값 공유 ★

이 스킬이 생성하는 `codes.ts` 는 **backflow:impl-error-codes** 가 생성하는 같은 이름의 파일과 **code 문자열 값이 완전히 일치** 해야 합니다. 파일 위치·enum 이름·언어는 달라도 무방하나, `code` 의 문자열 바이트는 동일해야 함. 예:

```typescript
// backend  (backflow:impl-error-codes 출력)
export const ErrorCode = { USER_NOT_FOUND: 'USER_NOT_FOUND', ... }

// frontend (이 스킬의 출력)
export const ErrorCode = { USER_NOT_FOUND: 'USER_NOT_FOUND', ... }
```

양방향 drift 는 `frontflow:validate-code` 의 새 룰이 감지합니다.

## 입력 TS 포맷

기대하는 §4 섹션 — backflow 쪽과 동일한 테이블 (backflow SKILL 참조).

### Grace period (v1.x)

TS 에 §4 가 없으면 **fallback**:
1. 경고: "TS-{ID}.md 에 §4 에러 코드 맵 섹션이 없습니다. specflow:generate-ts 재실행 권장."
2. 기본 5개 코드 생성 (backend 와 동일):
   - `AUTH_UNAUTHORIZED`, `AUTH_FORBIDDEN`, `VALIDATION_FAILED`, `SYSTEM_INTERNAL_ERROR`, `NETWORK_TIMEOUT`
3. UI flow 기본: 4xx → toast, 5xx → modal

## 출력

### 1. 에러 코드 enum (`frontend.md.error_handling.codes_file`, 기본 `src/errors/codes.ts`)

```typescript
// AUTO-GENERATED by frontflow:impl-error-handling from specs/TS-{ID}.md §4
// Code string values MUST match the backend's ErrorCode enum.
// Do not edit by hand. Re-run the skill to regenerate.

export const ErrorDomain = {
  AUTH: 'auth',
  USER: 'user',
  ORDER: 'order',
  SYSTEM: 'system',
} as const;
export type ErrorDomain = typeof ErrorDomain[keyof typeof ErrorDomain];

export const ErrorCode = {
  AUTH_INVALID_CREDENTIALS: 'AUTH_INVALID_CREDENTIALS',
  AUTH_TOKEN_EXPIRED: 'AUTH_TOKEN_EXPIRED',
  USER_NOT_FOUND: 'USER_NOT_FOUND',
  ORDER_PAYMENT_FAILED: 'ORDER_PAYMENT_FAILED',
  SYSTEM_EXTERNAL_TIMEOUT: 'SYSTEM_EXTERNAL_TIMEOUT',
} as const;
export type ErrorCode = typeof ErrorCode[keyof typeof ErrorCode];

export type UiFlow = 'inline' | 'toast' | 'modal' | 'redirect' | 'silent';

export interface ErrorMetaEntry {
  domain: ErrorDomain;
  httpStatus: number;
  i18nKey: string;
  retriable: boolean;
  cause?: ErrorCode;
  uiFlow?: UiFlow;
}

export const ErrorMeta: Record<ErrorCode, ErrorMetaEntry> = {
  [ErrorCode.AUTH_INVALID_CREDENTIALS]: {
    domain: ErrorDomain.AUTH,
    httpStatus: 401,
    i18nKey: 'errors.auth.invalid_credentials',
    retriable: false,
    uiFlow: 'inline',
  },
  // ... TS 행 그대로
};
```

### 2. 핸들러 파일 (`frontend.md.error_handling.handler_file`, 기본 `src/errors/handler.ts`)

```typescript
// AUTO-GENERATED by frontflow:impl-error-handling
import { ErrorCode, ErrorMeta, type UiFlow } from './codes';
// i18n_library 에 따라 import 경로 분기 (i18next / formatjs / lingui / inline)
import { t } from '@/i18n';

export type ErrorDecision = {
  code: ErrorCode | 'UNKNOWN_ERROR';
  message: string;
  uiFlow: UiFlow;
  retriable: boolean;
  field?: string;
  redirectTo?: string;
};

const DEFAULT_FLOW_BY_STATUS: Record<number, UiFlow> = {
  400: 'inline', 401: 'redirect', 403: 'toast', 404: 'toast',
  409: 'inline', 422: 'inline', 429: 'toast',
  500: 'modal', 502: 'modal', 503: 'modal', 504: 'modal',
};

export function handleError(raw: { code: string; message?: string; field?: string }): ErrorDecision {
  const known = (raw.code in ErrorMeta) ? (raw.code as ErrorCode) : null;
  if (!known) {
    console.warn('[error-handler] unknown code', raw.code);
    return {
      code: 'UNKNOWN_ERROR',
      message: t('errors.unknown') ?? '알 수 없는 오류가 발생했습니다',
      uiFlow: 'toast',
      retriable: false,
    };
  }
  const meta = ErrorMeta[known];
  return {
    code: known,
    message: t(meta.i18nKey),
    uiFlow: meta.uiFlow ?? DEFAULT_FLOW_BY_STATUS[meta.httpStatus] ?? 'toast',
    retriable: meta.retriable,
    field: raw.field,
    redirectTo: meta.uiFlow === 'redirect' ? inferRedirect(known) : undefined,
  };
}

function inferRedirect(code: ErrorCode): string | undefined {
  if (code === 'AUTH_TOKEN_EXPIRED' || code === 'AUTH_UNAUTHORIZED') return '/login';
  return undefined;
}
```

**순수 함수**. 렌더링 책임 없음. 실제 toast/modal/redirect 노출은 `impl-interactions` 또는 `impl-api-integration` 이 `ErrorDecision` 을 해석해서 수행.

### 3. UI 플로우 프리셋 (`frontend.md.error_handling.ui_flow_file`, 선택)

React 프로젝트에서 유용. 기본 toast/modal 컴포넌트가 디자인 시스템에 이미 있으면 스킵해도 됨.

```tsx
// AUTO-GENERATED by frontflow:impl-error-handling (optional preset)
import { toast } from '@/ui/toast';
import { openModal } from '@/ui/modal';
import { router } from '@/lib/router';
import type { ErrorDecision } from './handler';

export function presentError(decision: ErrorDecision): void {
  switch (decision.uiFlow) {
    case 'inline':
      // 호출 측에서 form field 에 직접 바인딩 (이 함수는 no-op 에 가까움)
      return;
    case 'toast':
      toast.error(decision.message);
      return;
    case 'modal':
      openModal({ title: '오류', body: decision.message, kind: 'error' });
      return;
    case 'redirect':
      if (decision.redirectTo) router.push(decision.redirectTo);
      return;
    case 'silent':
      return;
  }
}
```

### 4. i18n 리소스 (`i18n_library != 'inline'` 일 때만)

`frontend.md.error_handling.i18n_output_dir` 아래에 언어별 JSON 생성. backflow 와 **같은 스키마·같은 키**:

**`src/locales/errors.ko.json`**:
```json
{
  "errors": {
    "auth": { "invalid_credentials": "아이디 또는 비밀번호가 올바르지 않습니다" },
    "user": { "not_found": "사용자를 찾을 수 없습니다" },
    "unknown": "알 수 없는 오류가 발생했습니다"
  }
}
```

**공유 가능성**: backend 와 frontend 둘 다 같은 JSON 을 읽을 수도 있음 (모노레포 또는 공유 패키지). 별도 관리는 backflow·frontflow 각자 생성한 파일이 내용상 동일함에 의존 — validate-code drift 룰로 일관성 유지.

`i18n_library: inline` 시:
```typescript
[ErrorCode.USER_NOT_FOUND]: {
  // ...
  messageKo: '사용자를 찾을 수 없습니다',
  messageEn: 'User not found',
},
```
handler 가 `messageKo`/`messageEn` 을 현재 locale 에 맞게 선택.

### 5. 라이브러리별 핸들러 어댑터

- **i18next**: `import { t } from 'i18next'` + resources 등록 코드 제안
- **formatjs / react-intl**: `import { useIntl } from 'react-intl'` 훅 기반 래퍼 제안
- **lingui**: `import { t } from '@lingui/macro'`
- **inline**: handler 내부에서 `locale` 인자 받아 `messageKo`/`messageEn` 중 선택

첫 실행 시 사용자에게 라이브러리 선택 프롬프트 → `frontend.md.error_handling.i18n_library` 에 기록.

## 기존 파일과의 머지 전략

### Scenario A — 완전 신규 프론트엔드
3~4종 파일 모두 생성. 끝.

### Scenario B — 애드혹 에러 핸들링이 흩어진 프론트엔드

1. Grep 으로 기존 에러 분기 수집:
   - `catch (err)` 블록
   - `if (err.code === 'X')` 패턴
   - `toast.error(...)`, `alert(...)` 직접 호출
2. **`--dry-run`**: merge preview — 기존 분기 / 제안 `ErrorCode` / 제안 `uiFlow` 표 출력
3. 사용자 승인 후:
   - 신규 파일 생성
   - 기존 `catch` 블록에 `presentError(handleError(err))` 제안 (Edit tool, 각 변경마다 사용자 확인)

### Scenario C — 이미 표준화된 에러 핸들링이 있는 프론트엔드

1. `frontend.md.error_handling.codes_file` 경로 확인
2. 기존 파일이 backflow 와 같은 `code` 값을 사용하는지 검사
3. TS 에만 있고 기존 파일에 없는 code → append
4. 기존에만 있고 TS 에 없는 code → **수정하지 않음**, 사용자에게 TS 반영 권고

## 교차 검증

- `code` 문자열 값이 backflow 가 생성한 `codes.ts` 와 동일한지 확인 (backflow 파일이 접근 가능하면 대조; 아니면 TS 를 기준)
- `i18nKey` 가 `errors.{domain}.{snake}` 패턴 준수
- `uiFlow` 값이 허용 집합(`inline|toast|modal|redirect|silent`) 안에 있는지
- `handler.ts` 의 모든 code 분기가 `ErrorMeta` 의 키를 참조 (dead branch 0건)

## 멱등성

- AUTO-GENERATED 주석으로 자동 생성 파일 표시
- 재실행 시 사람이 편집한 영역 감지 → 중단 + 경고, `--force` 시 덮어쓰기
- i18n JSON 은 기존 non-errors 키(예: 프로젝트의 일반 번역) 를 덮어쓰지 않도록 주의 — `errors` 네임스페이스만 갱신, 다른 최상위 키 보존
