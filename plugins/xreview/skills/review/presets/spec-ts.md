# Technical Specification (TS) Review Preset

## Default Context

이 파일은 기술 명세서(Technical Specification, TS)입니다. 기능 명세(FS)를 기반으로 백엔드/프론트엔드 엔지니어가 작성한 API·데이터 모델·아키텍처 설계 문서입니다. 이 문서를 보고 구현팀이 코드를 작성하므로, 계약(contract)으로서 정확하고 완결적이어야 합니다.

## Review Perspective

### 1. API 계약 (API Contract)
- 모든 엔드포인트가 method·path·인증 요건·쿼리/바디 스키마·응답 스키마·상태 코드·에러 코드를 명시하는가
- 에러 코드 enum에 누락이 없는가 (FS의 모든 에러 상황이 매핑되는가)
- 페이지네이션·정렬·필터 규약이 일관되는가
- 버전 호환성·deprecation 전략이 있는가

### 2. 데이터 모델 (Data Model)
- 엔티티·필드·타입·nullable·기본값·유니크 제약·인덱스가 정의되었는가
- 엔티티 간 관계(FK, 카디널리티, 캐스케이드)가 명확한가
- 타임존·날짜·통화·식별자 표현이 일관되는가
- 소프트 삭제·이력 보관·감사 로그 정책이 서술되는가

### 3. 아키텍처 일관성 (Architecture)
- 레이어(컨트롤러·서비스·리포지토리) 책임 분리가 명확한가
- 의존 방향이 위반되는 부분이 있는가
- 외부 통합(큐·캐시·스토리지·3rd party API)의 인터페이스가 정의되는가
- 설정(config)·비밀(secret) 관리 방식이 있는가

### 4. 실패·복구 시나리오 (Failure & Recovery)
- 트랜잭션 경계가 명시되는가
- 멱등성(idempotency) 요구·키가 정의되는가
- 재시도·백오프·Dead Letter Queue 정책이 있는가
- 부분 실패 시 상태 일관성 복구 방안이 있는가
- 동시성(잠금·낙관적 제어·race condition) 처리가 서술되는가

### 5. FS와의 정합성 (FS Alignment)
- FS의 모든 기능 요구사항이 기술적으로 커버되는가
- FS의 비기능 요구사항(성능·가용성 등)이 설계에 반영되는가
- FS에 없는 과도한 기술 결정이 추가되지 않았는가

### 6. 확장성·성능 (Scalability & Performance)
- 예측 가능한 부하(TPS, 데이터 볼륨)에 대한 설계가 있는가
- 캐시·인덱스·페이지네이션 전략이 있는가
- 병목·확장 포인트가 식별되어 있는가

### 7. 보안·권한 (Security & Authorization)
- 인증·인가 모델이 엔드포인트별로 명시되는가
- 입력 검증·출력 인코딩 책임이 명확한가
- 민감 데이터의 저장·전송·로깅 정책이 있는가
- 레이트 리밋·남용 방지 메커니즘이 있는가

## Severity Guidance

- **critical**: API 계약 누락/모순, 트랜잭션 경계 오류, FS 요구사항 미커버, 보안 모델 부재
- **warning**: 멱등성·동시성·재시도 미정의, 인덱스 누락, 에러 코드 누락, 성능 설계 부재
- **info**: 명명 개선, 대안 설계 제안, 확장 고려 권고
