# meeting-doc-gen

spec-scanner와 impl-scanner의 분석 결과(중간 포맷)를 입력받아, **고정된 템플릿**에 맞춰 회의 준비 문서를 생성한다.

> 오케스트레이션은 meeting-orchestrator 에이전트가 담당한다. 이 스킬은 순수하게 "중간 포맷 → 최종 문서 변환"에만 집중한다.

## 입력

| 항목 | 필수 | 기본값 | 설명 |
|------|------|--------|------|
| topic | N | "전체" | 회의 주제/범위 |
| meeting_type | N | "기획 리뷰" | 회의 유형: 기획 리뷰 / 스프린트 리뷰 / 기술 결정 |
| output_path | N | `meeting-prep/` | 출력 파일 저장 경로 |

## 전제 조건

아래 중간 포맷 파일이 존재해야 한다:
- `{output_path}/spec-analysis.md` — spec-scanner 출력
- `{output_path}/impl-analysis.md` — impl-scanner 출력

파일이 없으면:
- 에이전트 경유 실행: 에러를 에이전트에 알림
- 단독 실행: 사용자에게 "먼저 spec-scanner / impl-scanner를 실행해주세요" 안내

## 처리 단계

### 1. 교차 분석

두 중간 포맷을 읽어서 아래를 도출한다:

**기획 검토 안건**:
- spec-analysis의 불명확 항목들을 우선순위별로 분류
- 정책 결정 필요 항목 (개발 블로커인 것 우선)
- 명세 구체화 필요 항목
- 문서 정합성 이슈

**구현 현황 리포트**:
- impl-analysis의 구현 상태 요약
- 기획 대비 구현 갭 (미구현 기능 중 중요도 높은 것)
- 기술적 논의 필요 사항

**종합 안건 분류**:
- 🔴 **긴급**: 현재 개발 진행을 막고 있는 항목
- 🟡 **중요**: 이번 스프린트 또는 가까운 시일 내 처리해야 할 항목
- 🟢 **참고**: 인지해두면 좋은 항목

**우선순위 산정 기준**:
1. 개발 블로커 여부 (다른 기능의 전제조건인가)
2. 일정 긴급도 (Phase 배정이 되어있고 임박한가)
3. 영향 범위 (다른 도메인/기능에 미치는 영향이 큰가)
4. 의사결정자 수 (기획/개발/운영 여러 직군이 관여하는가)

### 2. 문서 생성 (듀얼 출력)

교차 분석 결과를 **두 가지 형태**로 출력한다.

#### 출력 A: AI 친화적 문서 (상세 참조용)

`references/output-template.md` 템플릿을 따른다.

**출력 파일명**: `{output_path}/{주제}_{YYYY-MM-DD}.md`

**템플릿 적용 규칙**:
1. **ID 체계**: 각 섹션별 고유 prefix (P=정책, S=명세, D=정합성, G=갭, T=기술)
2. **파일 경로**: 반드시 상대 경로로, 실제 확인된 파일만 기록
3. **행 번호**: 가능한 경우 `(L{행})` 형태로 첨부
4. **체크리스트**: 종합 안건에서 "결정 사항"이 있는 것만 체크리스트로 추출
5. **빈 섹션 처리**: 해당 항목이 없으면 "해당 없음" 한 줄로 표시 (섹션 자체는 유지)
6. **회의 유형별 톤 조정**:
   - 기획 리뷰: 기획 검토 안건에 무게
   - 스프린트 리뷰: 구현 현황에 무게
   - 기술 결정: 기술 논의와 긴급 안건에 무게

#### 출력 B: 사람 친화적 문서 (회의용)

`references/output-template-human.md` 템플릿을 따른다.

**출력 파일명**: `{output_path}/{주제}_{YYYY-MM-DD}_회의요약.md`

**작성 원칙**:
1. **서술체 사용**: 테이블 위주가 아닌, 문장으로 상황과 결정 사항을 설명
2. **ID 코드 배제**: P-001, SPEC-003 같은 코드는 사용하지 않음. 자연어로만 표현
3. **파일 경로/행 번호 배제**: 기술적 참조 정보 제거. 필요시 "기획서의 유저 관리 섹션" 같은 자연어 표현
4. **한눈에 보기**: 문서 상단에 핵심 수치 요약 테이블 배치
5. **결정 중심**: "오늘 반드시 결정할 것" 섹션에서 각 안건마다 선택지(A/B)를 명확히 제시
6. **추정 소요 시간**: 각 긴급 안건에 개발 추정치를 붙여서 의사결정에 참고하도록 함
7. **개발 현황은 간결하게**: 영역별 한 줄 요약 + 진행 아이콘(✅🔧❌)으로 시각화
8. **회의 체크리스트**: 회의 중 바로 체크할 수 있는 자연어 문항 (ID 없음)

## 품질 기준

기존 수동 결과물의 핵심 특성을 유지한다:

1. **출처 추적**: 모든 주장에 `파일 경로:행 번호` 근거 제시
2. **현황-결정 쌍**: "지금 이렇다" + "뭘 결정해야 한다" 구조
3. **구체적 선택지 제시**: "결정 필요"에서 끝나지 않고, 가능한 옵션을 나열
4. **체크리스트 제공**: 회의에서 바로 사용 가능한 의사결정 목록

## 주의사항

1. **분석 결과에 의존**: 직접 파일을 읽지 않고, 스캐너들의 중간 포맷에서 정보를 추출함. 중간 포맷에 없는 정보를 지어내지 않을 것.
2. **과도한 나열 방지**: 불명확 항목이 많아도 전부 나열하지 않음. 우선순위 상위 항목을 종합 안건에서 정리.
3. **실행 가능한 안건**: "~가 불명확하다"로 끝나지 않고, "~를 결정해야 한다" 또는 "~에 대해 A/B/C 중 선택해야 한다"로 마무리.
4. **날짜 명시**: 분석 시점을 명확히 기록.
5. **기존 회의 결과 연동**: meeting-prep/에 이전 회의 문서가 있으면, 이전에 결정된 사항이 반영되었는지도 확인.
6. **두 문서의 역할 구분**: AI 문서는 후속 분석·자동화의 입력으로 쓰이고, 사람 문서는 회의 참석자가 인쇄하거나 화면으로 읽는 용도이다. 같은 정보를 담되, 표현 방식이 완전히 다르다.
