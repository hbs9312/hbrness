# 회의 준비 스킬 생성 컨텍스트

> 이 문서는 "회의 준비 자동화 스킬"을 만들기 위해 파악한 프로젝트 구조, 기획서 형식, 구현 현황, 기존 결과물 패턴을 정리한 것이다.

---

## 1. 프로젝트 레포지터리 구조

```
velvetalk/
├── docs/                          # 기획서·명세서 (기획자 관리)
│   ├── func-req/                  # 기능요구사항 (도메인별 폴더)
│   │   ├── chat/                  # 채팅창 관련 (19개 문서)
│   │   ├── studio/                # 스튜디오 관련 (content_create/ 하위 포함)
│   │   ├── global_layout/         # 글로벌 레이아웃 (8개 문서)
│   │   ├── adminpage/             # 관리자 페이지 (11개 문서)
│   │   └── managing_page/         # 매니징 페이지
│   ├── *.md                       # 통합 기술명세서 (v2_3 등), 백로그, 번역대조표
│   ├── CLAUDE.md                  # Claude 에이전트용 지침
│   └── README.md
├── backend/                       # Python FastAPI 백엔드
│   └── app/
│       ├── api/v1/endpoints/      # API 엔드포인트 (22개)
│       ├── models/                # SQLAlchemy 모델 (22개)
│       ├── repositories/          # DB 접근 레이어 (20개)
│       ├── schemas/               # Pydantic 스키마 (15개)
│       ├── services/              # 비즈니스 로직 (LLM, OAuth, Storage, Email 등)
│       ├── db/migrations/         # Alembic 마이그레이션 (40+개)
│       └── tests/                 # 테스트 (15개)
├── frontend/                      # React + TypeScript (Vite)
│   └── src/
│       ├── app/                   # 레이아웃, 프로바이더, i18n
│       ├── entities/              # 도메인 엔티티 (chat, content, viewer, chat-profile)
│       ├── features/              # 기능 단위 (chat-room, auth, admin, studio, banner 등)
│       └── ...
└── meeting-prep/                  # 회의 준비 자료 (스킬 출력 대상)
```

### 향후 프로젝트 추가 가능성
- `backend`, `frontend` 외에 새로운 프로젝트 디렉토리가 추가될 수 있음
- 스킬은 특정 프로젝트(벨벳톡)에 종속되지 않고, 범용적으로 동작해야 함

## 2. 기획서(docs) 형식 분석

### 2.1 디렉토리 구조 패턴
- 도메인별 폴더 (`func-req/{domain}/`)
- 각 폴더에 `00_인덱스.md` → 해당 도메인의 전체 기능 목록과 유저 플로우 정의
- 번호 prefix로 순서 지정 (`01_`, `02_`, ...)
- 루트에 통합 기술명세서 (`벨벳톡_기술명세서_v2_3.md` 등)

### 2.2 개별 기획서 문서 형식
- **마크다운 기반**, 일관된 구조:
  - `# 기능명` → `## 기능 정의` (한 줄 요약)
  - `## N. 세부 기능` → 테이블, 리스트, 코드블록으로 상세 기술
  - 조건부 로직은 테이블(`| 구간 | 조건 | 동작 |`) 형태
  - 다른 문서 참조는 `→ [파일명](./파일명.md)` 형식
  - 미구현/후속 항목은 취소선(~~) 또는 별도 표기 ("정식 출시 이후")
  - `> 블록인용`으로 보충 설명, 주의사항

### 2.3 기획서에서 나타나는 "불명확한 부분" 패턴
- **"미정"**, **"백로그"**, **"TBD"** 명시
- **취소선(~~)으로 표기된 미구현 항목** — 우선순위/일정 미확정
- **"결정 필요"** 라벨이 붙은 항목
- 조건부 로직에서 **경계값이 비어있는 경우** (예: 임계값 미지정)
- 다른 문서 참조가 있으나 **해당 문서가 존재하지 않는 경우**
- API/기술적 구현 방법이 **"필요"로만 기록**되고 구체 방안이 없는 경우
- Phase 배정이 안 된 항목
- **99_미구현_및_확인필요_항목.md** 같은 전용 파일에 모은 것들

## 3. 구현 프로젝트 현황 파악 방법

### 3.1 Backend (Python FastAPI)
- **기술 스택**: FastAPI, SQLAlchemy, Alembic, Pydantic
- **구조**: endpoint → service → repository → model 계층 구조
- **구현 현황 파악 포인트**:
  - `app/api/v1/endpoints/` — 어떤 API가 존재하는지 (파일명 = 도메인)
  - `app/models/` — 어떤 DB 모델이 정의되었는지
  - `app/services/` — 비즈니스 로직 구현 범위
  - `app/db/migrations/versions/` — 최근 마이그레이션으로 진행 상황 추적
  - `app/tests/` — 어떤 기능에 테스트가 있는지
  - `git log --oneline -20` — 최근 커밋으로 작업 동향 파악

### 3.2 Frontend (React + TypeScript + Vite)
- **기술 스택**: React, TypeScript, Vite, Zustand(store), i18n
- **구조**: FSD(Feature-Sliced Design) 패턴
  - `entities/` — 도메인 데이터 모델과 API
  - `features/` — 기능 단위 UI + 로직
  - `app/` — 글로벌 레이아웃, 프로바이더
- **구현 현황 파악 포인트**:
  - `src/features/` 디렉토리 목록 — 어떤 feature가 구현되었는지
  - `src/entities/` — 어떤 도메인 엔티티가 정의되었는지
  - 각 feature의 `api/`, `model/`, `ui/` 하위 파일 — 구현 깊이
  - `git log --oneline -20` — 최근 작업 동향

### 3.3 범용적 구현 현황 파악 전략
프로젝트 성격에 관계없이 적용 가능한 방법:
1. **디렉토리 트리 스캔** — 전체 구조 파악
2. **git log** — 최근 작업 흐름
3. **주요 설정 파일** — package.json, requirements.txt, pyproject.toml 등에서 의존성 확인
4. **README / SETUP 파일** — 프로젝트 개요
5. **테스트 파일** — 구현 완료 기능 힌트

## 4. 기존 meeting-prep 결과물 분석

### 4.1 결과물 형식 (`이미지_검열_명세_검토.md`)
```
# {주제} ({날짜})
> 회의 목적: {한 줄 요약}

## 1. 현재 명세된 내용
- 테이블 형태로 정리 (| 명세된 항목 | 내용 | 파일 |)
- 파일 경로와 행 번호까지 정확히 명시

## 2. 누락된 항목 및 검토 대상 파일
- 항목별로:
  - **검토 파일**: 해당 기획서 경로
  - **현황**: 현재 어떻게 되어있는지
  - **결정 필요**: 회의에서 무엇을 결정해야 하는지

## 3. 회의 시 결정 체크리스트
- [ ] 체크박스 형태로 의사결정 항목 나열
```

### 4.2 결과물의 핵심 특성
- **출처 추적**: 모든 주장에 파일 경로 + 행 번호 근거 제시
- **현황-결정 쌍**: "지금 이렇다" + "뭘 결정해야 한다" 구조
- **구체적 선택지 제시**: 단순히 "결정 필요"가 아니라 가능한 옵션을 나열
- **체크리스트**: 회의에서 바로 사용 가능한 의사결정 목록

## 5. 스킬이 수행해야 할 작업 흐름 (초안)

### 입력
- `docs/` 디렉토리 (기획서/명세서)
- 구현 프로젝트 디렉토리들 (`backend/`, `frontend/`, 향후 추가 가능)
- (선택) 특정 주제/범위 지정

### 처리
1. **기획서 스캔**
   - 모든 기획서 파일을 읽고 구조 파악
   - 불명확 항목 탐지: "미정", "TBD", "백로그", 취소선, "결정 필요", 빈 값 등
   - 문서 간 참조 무결성 검사 (참조된 파일이 실제로 존재하는지)
   - 기술적 구현 방법이 빠진 항목 식별

2. **구현 현황 분석**
   - 각 프로젝트의 디렉토리 구조 스캔
   - git log로 최근 작업 동향 파악
   - 기획서 기능 vs 실제 구현 매핑 (어떤 기능이 구현되었고, 어떤 것이 아직인지)

3. **종합 분석 및 회의 자료 생성**
   - 기획서 불명확 항목 → 회의 안건
   - 기획-구현 갭 → 우선순위 논의 안건
   - 구현 중 발견된 기술적 이슈 → 기술 결정 안건
   - 파일 경로 + 행 번호 근거 첨부

### 출력
- `meeting-prep/{주제}_{날짜}.md` 형태의 회의 준비 문서

## 6. 스킬 설계 시 고려사항

### 6.1 범용성
- 벨벳톡 프로젝트에 특화되지 않아야 함
- 기획서 형식이 마크다운이라는 것 외에 특정 포맷을 가정하지 않아야 함
- 프로젝트 기술 스택에 무관하게 구현 현황을 파악할 수 있어야 함

### 6.2 스캔 범위 제어
- 전체 스캔 vs 특정 도메인/주제 한정 스캔 선택 가능해야 함
- 기획서가 많을 경우 컨텍스트 윈도우 한계 고려

### 6.3 출력 품질
- 단순 나열이 아니라, 회의에서 바로 사용 가능한 수준의 구조화
- 의사결정 체크리스트 필수
- 모든 주장에 출처(파일 경로) 근거 제시

### 6.4 기획서 불명확 탐지 기준
| 탐지 패턴 | 설명 |
|-----------|------|
| "미정", "TBD", "TODO" | 명시적 미결 표시 |
| 취소선 (`~~텍스트~~`) | 미구현/보류 항목 |
| "정식 출시 이후", "Phase N 이후" | 일정 미확정 |
| "결정 필요", "확인 필요" | 명시적 결정 요청 |
| "~ 필요" (API 필요, 검토 필요 등) | 구체 방안 없는 요구 |
| 참조 파일 미존재 | 문서 간 참조 깨짐 |
| 빈 셀/빈 값의 테이블 | 정보 누락 |
| 99_미구현_*.md 류 파일 | 전용 미결 사항 모음 |

### 6.5 기획-구현 매핑 전략
- 기획서의 도메인 키워드 ↔ 프로젝트의 파일명/디렉토리명 매칭
- 예: 기획서 `chat/` 도메인 → backend `endpoints/chats.py`, `services/chat_service.py`, frontend `features/chat-room/`
- 기획서에 정의된 기능 목록 vs 실제 endpoint/component 존재 여부 비교

## 7. 스킬에 필요한 사용자 입력 (실행 시)

스킬 실행 시 아래 정보를 사용자에게 확인해야 함:

1. **기획서 경로**: docs 디렉토리 위치 (기본: `docs/`)
2. **프로젝트 디렉토리 목록**: 구현 프로젝트들 (기본: `backend/`, `frontend/`)
3. **회의 주제/범위**: 전체 스캔 or 특정 도메인 한정 (예: "이미지 검열", "채팅 기능")
4. **출력 위치**: 회의 준비 문서 저장 경로 (기본: `meeting-prep/`)
5. **회의 유형**: 기획 리뷰 / 스프린트 회고 / 기술 결정 등 (출력 톤에 영향)
