# meeting-doc-gen 스킬 생성 컨텍스트

> 이 문서는 skill-creator에게 전달하여 meeting-doc-gen 스킬을 생성하기 위한 컨텍스트이다.

---

## 1. 스킬 목적

spec-scanner와 impl-scanner의 분석 결과(중간 포맷)를 입력받아, **고정된 템플릿**에 맞춰 회의 준비 문서를 생성하는 스킬.

> **역할 변경 (v2)**: 이전에는 이 스킬이 오케스트레이터를 겸했으나, 이제 오케스트레이션은 **meeting-orchestrator 에이전트**가 담당한다. 이 스킬은 순수하게 "중간 포맷 → 최종 문서 변환"에만 집중한다.

---

## 2. 트리거 조건

### 에이전트 경유 (주 사용 경로)
- meeting-orchestrator 에이전트가 spec-analysis.md와 impl-analysis.md를 준비한 뒤, 이 스킬을 호출하여 최종 문서를 생성

### 단독 사용 (보조)
- 이미 spec-analysis.md, impl-analysis.md가 존재하는 상태에서 "회의 문서만 다시 생성해줘"
- 중간 포맷을 수동으로 편집한 후 문서 재생성

---

## 3. 입력

| 항목 | 필수 | 기본값 | 설명 |
|------|------|--------|------|
| docs_path | Y | - | 기획서 디렉토리 경로 |
| project_paths | Y | - | 구현 프로젝트 디렉토리 경로 리스트 |
| topic | N | "전체" | 회의 주제/범위 (전체 or 특정 주제) |
| meeting_type | N | "기획 리뷰" | 회의 유형: 기획 리뷰 / 스프린트 리뷰 / 기술 결정 |
| output_path | N | `meeting-prep/` | 출력 파일 저장 경로 |

---

## 4. 처리 단계

### 4.1 사용자 확인 (실행 시)

스킬이 트리거되면 먼저 사용자에게 아래를 확인한다:
1. 회의 주제/범위 — "전체 점검인지, 특정 주제(예: 이미지 검열, 채팅 기능)인지"
2. 기획서 경로와 프로젝트 경로 — "기본 경로로 진행할지, 다른 경로가 있는지"
3. 회의 성격 — "기획 리뷰인지, 스프린트 리뷰인지, 기술 결정 회의인지"

### 4.2 입력 파일 확인

중간 포맷 파일이 존재하는지 확인한다:
- `{output_path}/spec-analysis.md` — spec-scanner 출력
- `{output_path}/impl-analysis.md` — impl-scanner 출력

> 이 파일들은 meeting-orchestrator 에이전트가 서브에이전트를 통해 미리 생성해둔다. meeting-doc-gen은 스캐너를 직접 실행하지 않는다.

파일이 없으면:
- 에이전트 경유 실행인 경우: 에러 — 에이전트에 알림
- 단독 실행인 경우: 사용자에게 "먼저 spec-scanner / impl-scanner를 실행해주세요" 안내

### 4.3 교차 분석

두 중간 포맷을 읽어서 아래를 도출한다:

**기획 검토 안건**:
- spec-analysis의 불명확 항목들을 우선순위별로 분류
- 정책 결정 필요 항목 (개발 블로커인 것 우선)
- 명세 구체화 필요 항목
- 문서 정합성 이슈

**구현 현황 리포트**:
- impl-analysis의 구현 상태 요약
- 기획 대비 구현 갭 (미구현 기능 중 중요도 높은 것)
- 기술적 논의 필요 사항

**종합 안건 분류** (핵심 로직):
- 🔴 **긴급**: 현재 개발 진행을 막고 있는 항목
  - 기획 미확정인데 구현 일정이 잡혀있는 것
  - API 스키마/네이밍 불일치로 프론트-백 연동 불가인 것
- 🟡 **중요**: 이번 스프린트 또는 가까운 시일 내 처리해야 할 항목
  - 다음 Phase에 영향을 미치는 미결정 사항
  - 부분 구현 상태에서 방향 결정이 필요한 것
- 🟢 **참고**: 인지해두면 좋은 항목
  - 백로그 항목
  - 장기 계획 관련 결정

**우선순위 산정 기준**:
1. 개발 블로커 여부 (다른 기능의 전제조건인가)
2. 일정 긴급도 (Phase 배정이 되어있고 임박한가)
3. 영향 범위 (다른 도메인/기능에 미치는 영향이 큰가)
4. 의사결정자 수 (기획/개발/운영 여러 직군이 관여하는가)

### 4.4 문서 생성

고정 템플릿에 맞춰 최종 문서를 생성한다. (템플릿은 아래 섹션 5 참조)

---

## 5. 출력 템플릿 (고정)

이 템플릿은 변경되지 않는다. 모든 회의 준비 문서는 이 구조를 따른다.

```markdown
# {회의 주제} ({YYYY-MM-DD})

> 회의 목적: {한 줄 요약}
> 생성 기준: spec-analysis ({날짜}), impl-analysis ({날짜})

---

## 1. 기획 검토 안건

### 1.1 정책 결정 필요 항목

개발 진행 전 기획/운영 측에서 결정해야 하는 항목.

| # | 항목 | 현황 | 결정 필요 사항 | 관련 파일 |
|---|------|------|--------------|----------|
| P-001 | {항목명} | {현재 상태} | {무엇을 결정해야 하는지} | `{파일 경로}` (L{행}) |

### 1.2 명세 불명확 항목

명세서에서 구체화가 필요한 항목.

| # | 항목 | 현황 | 구체화 필요 사항 | 관련 파일 |
|---|------|------|----------------|----------|
| S-001 | {항목명} | {현재 상태} | {무엇을 구체화해야 하는지} | `{파일 경로}` (L{행}) |

### 1.3 문서 정합성 이슈

기획서 간 또는 기획서-코드 간 불일치.

| # | 이슈 | 상세 | 관련 파일 |
|---|------|------|----------|
| D-001 | {이슈 제목} | {상세 설명} | `{파일A}`, `{파일B}` |

---

## 2. 구현 현황 리포트

### 2.1 최근 진행 상황

{프로젝트별 2~3줄 요약}

### 2.2 기획 대비 구현 갭

| # | 기획 기능 | 기획서 | 구현 상태 | 비고 |
|---|----------|--------|----------|------|
| G-001 | {기능명} | `{기획서 파일}` | ❌ 미구현 / 🔧 부분 구현 | {참고} |

### 2.3 기술적 논의 필요 사항

구현 과정에서 발견된 기술적 이슈나 설계 결정이 필요한 사항.

| # | 항목 | 상세 | 선택지 |
|---|------|------|--------|
| T-001 | {항목명} | {상세 설명} | {가능한 선택지 나열} |

---

## 3. 종합 안건 (우선순위순)

### 🔴 긴급 (블로커)

현재 개발을 막고 있거나 즉시 결정이 필요한 사항.

1. **{안건 제목}** — {한 줄 설명}
   - 배경: {왜 긴급한지}
   - 결정 사항: {무엇을 결정해야 하는지}
   - 관련: {관련 항목 ID들, 예: P-001, G-002}

### 🟡 중요 (이번 스프린트)

가까운 시일 내 처리해야 할 사항.

1. **{안건 제목}** — {한 줄 설명}
   - 배경: {왜 중요한지}
   - 결정 사항: {무엇을 결정해야 하는지}

### 🟢 참고 (백로그)

인지해두면 좋은 사항. 즉각적인 결정은 불필요.

1. **{안건 제목}** — {한 줄 설명}

---

## 4. 회의 시 결정 체크리스트

- [ ] {결정 항목 1} (← {관련 안건 ID})
- [ ] {결정 항목 2} (← {관련 안건 ID})
- [ ] {결정 항목 3} (← {관련 안건 ID})
...
```

### 템플릿 적용 규칙

1. **ID 체계**: 각 섹션별 고유 prefix (P=정책, S=명세, D=정합성, G=갭, T=기술)
2. **파일 경로**: 반드시 상대 경로로, 실제 확인된 파일만 기록
3. **행 번호**: 가능한 경우 `(L{행})` 형태로 첨부
4. **체크리스트**: 섹션 3의 안건에서 "결정 사항"이 있는 것만 체크리스트로 추출
5. **빈 섹션 처리**: 해당 항목이 없으면 "해당 없음" 한 줄로 표시 (섹션 자체는 유지)
6. **회의 유형별 톤 조정**:
   - 기획 리뷰: 기획 검토 안건(섹션 1)에 무게
   - 스프린트 리뷰: 구현 현황(섹션 2)에 무게
   - 기술 결정: 기술 논의(섹션 2.3)와 긴급 안건(섹션 3 🔴)에 무게

---

## 6. 기존 결과물 참고

기존에 수동으로 작성된 `이미지_검열_명세_검토.md`의 핵심 특성을 이 스킬에서도 유지해야 한다:

1. **출처 추적**: 모든 주장에 `파일 경로:행 번호` 근거 제시
2. **현황-결정 쌍**: "지금 이렇다" + "뭘 결정해야 한다" 구조
3. **구체적 선택지 제시**: "결정 필요"에서 끝나지 않고, 가능한 옵션을 나열
4. **체크리스트 제공**: 회의에서 바로 사용 가능한 의사결정 목록

---

## 7. 주의사항

1. **분석 결과에 의존**: meeting-doc-gen은 직접 파일을 읽지 않고, 스캐너들의 중간 포맷에서 정보를 추출함. 중간 포맷에 없는 정보를 지어내지 않을 것.
2. **과도한 나열 방지**: 불명확 항목이 50개라도 전부 나열하지 않음. 우선순위 상위 항목을 섹션 3에서 종합하고, 나머지는 섹션 1/2 테이블에서 참조.
3. **실행 가능한 안건**: "~가 불명확하다"로 끝나지 않고, "~를 결정해야 한다" 또는 "~에 대해 A/B/C 중 선택해야 한다"로 마무리.
4. **날짜 명시**: 분석 시점을 명확히 기록. 기획서/코드는 계속 변경되므로 시점이 중요.
5. **기존 회의 결과 연동**: meeting-prep/ 에 이전 회의 문서가 있으면, 이전에 결정된 사항이 반영되었는지도 확인.
