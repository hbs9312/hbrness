# 에러 코드·HTTP 매핑·i18n 리소스 생성 (B-Err)

ultrathink

당신은 백엔드 개발자입니다.
기술 명세서(TS)의 §4 에러 코드 맵을 코드 상수와 HTTP 매핑, 다국어 리소스로 변환합니다.

## 컨텍스트 로드

- **프로젝트 설정**: [backend.md](../../context/backend.md) — `error_handling` 섹션 **필수**
- **설계 문서**: `docs/designs/phase-1-error-contract.md` (프로젝트 루트에 있으면 참조)
- **태스크-파일 맵**: `.backflow/task-file-map.md` (있으면) — 이 스킬이 담당하는 파일 목록

## 입력

$ARGUMENTS 에서:
1. **기술 명세서(TS)** → `§4 에러 코드 맵` 섹션 (필수). `specflow:generate-ts` 4-tier 포맷 이후 생성된 TS
2. **기능 명세서(FS)** (선택) → `§도메인 정의` 섹션 (domain 교차검증용)

## 태스크-파일 맵 준수

`.backflow/task-file-map.md`가 존재하면:
1. `impl_skill: impl-error-codes` 항목만 필터
2. `action: create` → 해당 경로에 새 파일 생성
3. `action: modify` → 기존 파일에 행 append (기존 코드 보존)
4. 맵에 없는 파일은 생성하지 않음

맵이 없으면 `backend.md.error_handling.*` 경로를 기준으로 독자 판단.

## ★ 실행 위치 — impl-services 직전 ★

이 스킬은 map-tasks 직후, **impl-services 보다 먼저** 실행되는 것이 이상적입니다.
서비스 계층이 throw 할 `ErrorCode` enum 상수가 미리 존재해야 서비스 코드 품질이 좋아집니다.

실행 순서:
```
map-tasks → impl-schema → impl-repositories → impl-error-codes → impl-services → impl-controllers → impl-middleware → ...
```

impl-middleware 는 이 스킬이 만든 `HTTP_STATUS_MAP` 을 **import 해서 사용** 합니다. impl-middleware 가 직접 매핑 테이블을 만들지 않도록 주의 (중복 금지).

## ★ 계약: TS 가 source of truth ★

TS §4 에러 코드 맵 의 행 수 = 생성 파일의 ErrorCode 엔트리 수. 양방향 drift 는 `backflow:validate-code` 의 새 룰이 감지합니다.

- TS 에만 있고 코드에 없음 → missing (이 스킬이 append)
- 코드에만 있고 TS 에 없음 → orphan (사용자에게 경고, 수동 제거 요청)

## 입력 TS 포맷

기대하는 §4 섹션 포맷 (`specflow:generate-ts` 템플릿):

```markdown
# 4. 에러 코드 맵
| domain | code | http_status | i18n_key | message_ko | message_en | retriable |
|---|---|---|---|---|---|---|
| auth | AUTH_INVALID_CREDENTIALS | 401 | errors.auth.invalid_credentials | 아이디 또는 비밀번호가 올바르지 않습니다 | Invalid credentials | false |
| user | USER_NOT_FOUND | 404 | errors.user.not_found | 사용자를 찾을 수 없습니다 | User not found | false |
```

### Grace period (v1.x)

TS 에 §에러 코드 맵이 없으면 **fallback**:
1. 경고 출력: "TS-{ID}.md 에 §4 에러 코드 맵 섹션이 없습니다. specflow:generate-ts 재실행 권장."
2. 기본 5개 코드 생성:
   - `AUTH_UNAUTHORIZED` (401, errors.auth.unauthorized)
   - `AUTH_FORBIDDEN` (403, errors.auth.forbidden)
   - `VALIDATION_FAILED` (400, errors.validation.failed)
   - `SYSTEM_INTERNAL_ERROR` (500, errors.system.internal_error)
   - `NETWORK_TIMEOUT` (504, errors.network.timeout)
3. TODO 주석 추가: 실제 프로젝트 코드는 TS 에서 생성돼야 함

## 출력

### 1. 에러 코드 enum (`backend.md.error_handling.error_code_enum`, 기본 `src/errors/codes.ts`)

언어별 출력 형태:

**TypeScript (NestJS/Express/Fastify)**:
```typescript
// AUTO-GENERATED by backflow:impl-error-codes from specs/TS-{ID}.md §4
// Do not edit by hand. Re-run the skill to regenerate.

export const ErrorDomain = {
  AUTH: 'auth',
  USER: 'user',
  ORDER: 'order',
  SYSTEM: 'system',

export const ErrorCode = {
  AUTH_INVALID_CREDENTIALS: 'AUTH_INVALID_CREDENTIALS',
  AUTH_TOKEN_EXPIRED: 'AUTH_TOKEN_EXPIRED',

  ORDER_PAYMENT_FAILED: 'ORDER_PAYMENT_FAILED',

export type ErrorCode = typeof ErrorCode[keyof typeof ErrorCode];

export interface ErrorMetaEntry {

export const ErrorMeta: Record<ErrorCode, ErrorMetaEntry> = {

  },
  // ... TS 행 그대로

```

### 2. HTTP 매핑 파일 (`backend.md.error_handling.http_mapping_file`, 기본 `src/errors/http-mapping.ts`)

```typescript
// AUTO-GENERATED by backflow:impl-error-codes
import { ErrorCode, ErrorMeta } from './codes';

export const HTTP_STATUS_MAP: Record<ErrorCode, number> = Object.fromEntries(
  Object.entries(ErrorMeta).map(([code, meta]) => [code, meta.httpStatus])
) as Record<ErrorCode, number>;
```

(단순 derived table이므로 ErrorMeta 를 런타임에 재구성해도 되고 빌드 시 펼쳐도 됨. 프로젝트 선호에 따름.)

### 3. i18n 리소스 (`i18n_enabled: true` 일 때만)

`backend.md.error_handling.i18n_output_dir` 아래에 언어별 JSON 생성:

**`src/locales/errors.ko.json`**:
```json
{
  "errors": {
    "auth": {
      "invalid_credentials": "아이디 또는 비밀번호가 올바르지 않습니다",
      "token_expired": "로그인이 만료되었습니다. 다시 로그인해주세요"
    },
    "user": {
      "not_found": "사용자를 찾을 수 없습니다"
    }
  }
}
```

**`src/locales/errors.en.json`**: 동일 구조, message_en 컬럼 값 사용.

`languages` 배열에 `ko`/`en` 외 언어가 있으면 TS 테이블에 해당 컬럼이 있어야 함 (없으면 warning, 해당 언어 파일은 생성 스킵).

`i18n_enabled: false` 일 때는 ErrorMeta 에 `messageKo`/`messageEn` 필드를 직접 포함:

```typescript
[ErrorCode.USER_NOT_FOUND]: {
  domain: ErrorDomain.USER,
  httpStatus: 404,
  messageKo: '사용자를 찾을 수 없습니다',
  messageEn: 'User not found',
  retriable: false,
},
```

## 기존 파일과의 머지 전략

### Scenario A — 완전 신규 프로젝트
대상 파일 3종 모두 생성. 끝.

### Scenario B — 애드혹 에러 상수가 흩어진 프로젝트

1. `.backflow/service-registry.md` 또는 `Grep` 으로 기존 "에러 후보" 수집:
   - `throw new Error('...')` 의 메시지들
   - `throw new HttpException(...)` / `throw new NotFoundException(...)` 등 framework exception
   - 이미 enum 비슷하게 정의된 것 (`export enum ErrorCodes ...`)
2. **`--dry-run`**: merge preview 표 출력
   - 열: 기존 사용처 / 제안 ErrorCode / 제안 http_status / diff 규모
3. 사용자 승인 후 실행:
   - 신규 파일 생성
   - 기존 `throw` 호출처에 codemod 제안 (Edit tool, 각 변경마다 사용자 확인)
4. 미적용 호출처는 TODO 주석으로 표시

codemod 는 과도히 공격적이지 않게. 한 세션 내 10회 이상 변경이면 배치로 묶어 요약 제시.

### Scenario C — 이미 표준화된 에러 체계

1. `backend.md.error_handling.error_code_enum` 이 지정된 파일 존재 확인
2. 파일 패턴 분석:
   - enum + HTTP mapping + i18n key 삼박자가 있음 → "표준 준수"
   - 일부만 있음 → 누락 부분 보강 제안
3. TS 에만 있고 기존 파일에 없는 code → append (기존 포맷 보존)
4. 기존 파일에만 있고 TS 에 없는 code → **이 스킬은 수정하지 않음**. 사용자에게 "TS 에 반영하거나 코드에서 제거" 경고만.
5. `specflow:revise` 를 제안해 TS 쪽을 맞추도록 유도

## 교차 검증

- domain 열이 FS §도메인 정의와 일치하는지 확인. 불일치 시 warning + 행별 출력.
- code 전역 유일성 검사 (중복 행 감지 → error, 생성 중단).
- http_status 가 허용 범위(400/401/403/404/409/422/429/500/502/503/504)인지 확인. 그 외는 warning + 행별 출력.
- i18n_key 패턴 `errors.{domain}.{snake}` 준수 확인. 불일치 시 warning.

## 멱등성

이 스킬은 여러 번 실행해도 같은 입력에 대해 같은 출력이 나와야 합니다:
- 파일 상단 `// AUTO-GENERATED` 주석으로 자동 생성 파일 표시
- 재실행 시 기존 파일의 사람이 편집한 영역(주석·커스텀 함수)은 **감지하면 중단 + 경고**, 사용자가 `--force` 를 주면 덮어쓰기
- 자동 생성 영역은 재생성 시마다 전체 재작성
