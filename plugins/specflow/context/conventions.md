# 문서 컨벤션

이 파일은 모든 specflow 스킬이 참조하는 공통 규칙입니다.
프로젝트별로 수정하지 마세요. 프로젝트별 용어는 `glossary.md`에 정의합니다.

## 문서 ID 체계

| 문서 유형 | 접두사 | 형식 | 예시 |
|-----------|--------|------|------|
| 기능 명세서 | FS | FS-{YYYY}-{NNN} | FS-2026-001 |
| 와이어프레임 | WF | WF-{YYYY}-{NNN} | WF-2026-001 |
| 기술 명세서 | TS | TS-{YYYY}-{NNN} | TS-2026-001 |
| 화면설계서 | UI | UI-{YYYY}-{NNN} | UI-2026-001 |
| 테스트 명세서 | QA | QA-{YYYY}-{NNN} | QA-2026-001 |

## 참조 ID 체계

| 참조 유형 | 접두사 | 소속 문서 |
|-----------|--------|----------|
| 사용자 스토리 | US-{NNN} | 기능 명세서 |
| 비즈니스 룰 | BR-{NNN} | 기능 명세서 |
| 기능 요구사항 | FR-{NNN} | 기능 명세서 |
| 수용 기준 | AC-{NNN} | 기능 명세서 |
| 아키텍처 결정 | ADR-{NNN} | 기술 명세서 |
| 테스트 케이스 | TC-{NNN} | 테스트 명세서 |
| 태스크 | TASK-{NNN} | 구현 계획 |

## 문서 상태값

문서 상태: `Draft` → `In Review` → `Approved` → `Deprecated`

## 메타데이터 필수 필드

모든 문서의 헤더에 반드시 포함:

```yaml
문서 ID: {접두사}-{YYYY}-{NNN}
작성일: {YYYY-MM-DD}
상태: {Draft | In Review | Approved | Deprecated}
작성자: {역할명}
리뷰어: {역할명 목록}
```

참조 문서가 있는 경우 추가:

```yaml
기능 참조: FS-{YYYY}-{NNN}
기술 참조: TS-{YYYY}-{NNN}
```

## 출력 형식 규칙

- 모든 산출물은 YAML 코드블록 안에 작성
- 섹션 구분은 `# ─────` 코멘트 라인 사용
- 참조 ID는 반드시 `{접두사}-{NNN}` 형식으로 인라인 표기
  예: "화자명 중복 불가 (BR-003)"
