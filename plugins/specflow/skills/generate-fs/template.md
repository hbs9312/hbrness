# 기능 명세서 출력 템플릿

모든 {placeholder}를 실제 내용으로 교체하세요.

---

```yaml
문서 ID: FS-{YYYY}-{NNN}
기능명: {기능의 공식 명칭}
작성일: {YYYY-MM-DD}
상태: Draft
작성자: PM
리뷰어: 디자이너, 테크리드

# ─────────────────────────────────────
# 1. 배경과 목적
# ─────────────────────────────────────

## 문제 정의
{현재 상태의 문제점과 정량적 근거}

## 목표 지표
- {지표 1}: {목표 수치} ({측정 방법})
- {지표 2}: {목표 수치} ({측정 방법})

# ─────────────────────────────────────
# 2. 사용자 스토리
# ─────────────────────────────────────

## US-001: {스토리 제목}
AS A {역할}
I WANT TO {원하는 행동}
SO THAT {기대하는 결과}

# ─────────────────────────────────────
# 3. 비즈니스 룰
# ─────────────────────────────────────

BR-001: {구체적 수치를 포함한 비즈니스 룰}
BR-002: {구체적 수치를 포함한 비즈니스 룰}

# ─────────────────────────────────────
# 4. UI/UX 흐름
# ─────────────────────────────────────

## 진입 경로
- {메인 경로}: {설명}

## 4-1. {화면명}
주요 요소:
- {요소 1}: {역할과 데이터}

사용자 액션:
- {액션 1} → {결과}

상태:
- 정상: {설명}
- 로딩: {설명}
- 에러: {설명 + 복구 경로}
- 빈 상태: {설명 + 행동 유도}

# ─────────────────────────────────────
# 5. 수용 기준 (Acceptance Criteria)
# ─────────────────────────────────────

AC-001: {입력/조건} → {기대결과} (관련: US-001)
AC-002: {입력/조건} → {기대결과} (관련: BR-001 위반 시)

# ─────────────────────────────────────
# 6. 범위 밖 (Out of Scope)
# ─────────────────────────────────────

- {제외 기능 1} ({제외 이유 또는 예정 Phase})
```
