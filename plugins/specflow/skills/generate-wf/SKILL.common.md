# 와이어프레임 생성 (G2)

당신은 PM → 디자이너 핸드오프를 위한 와이어프레임을 작성합니다.

## 공통 컨텍스트 로드

- **문서 컨벤션**: [conventions.md](../../context/conventions.md)
- **도메인 용어집**: [glossary.md](../../context/glossary.md)

## 입력

$ARGUMENTS 로 전달된 기능 명세서(FS) 파일을 Read로 읽으세요.
기능 명세서의 상태가 `Approved`인지 확인하세요.

## 모드 판별

| 조건 | 모드 | 동작 |
|------|------|------|
| `--base` 인자 없음 | **full** (기본) | 전체 WF를 새로 생성 |
| `--base [기존 WF 경로]` 있음 | **incremental** | 기존 WF를 보존하고 새로 추가된 FS 항목에 대한 화면만 생성하여 병합 |

### incremental 모드 규칙

1. 기존 WF 파일을 Read로 읽고 현재 화면 목록과 참조 ID 매핑을 파악
2. FS에서 기존 WF에 매핑되지 않은 **새 항목**(US, BR, AC)을 식별
3. 새 항목에 대해서만 화면/상태를 생성
4. 기존 화면은 원문 그대로 보존 — 내용 변경 금지
5. 새 화면 ID는 기존 시퀀스 이어서 할당
6. 화면 전환 맵에 새 경로 추가 (기존 화면 ↔ 새 화면 양방향)
7. 상태 매트릭스에 새 화면 행 추가
8. 기존 화면이 새 항목 때문에 전환 경로가 추가되면 해당 부분만 수정

## 핵심 원칙: "어디에, 어떤 구조로" — 시각 디자인은 포함하지 않음

### 허용: 레이아웃, 요소 위치, 정보 계층, 상태 전환, 버튼 활성/비활성 조건
### 금지: 색상, 폰트, 애니메이션 수치, 디자인 토큰

## 생성 전략

1. 기능 명세의 UI/UX 흐름 섹션을 화면 단위로 파싱
2. 각 화면에서 BR을 UI 요소로 매핑
3. AC에서 에러 상태를 추출하여 상태 매트릭스에 반영
4. 화면 전환 맵: 정방향 → 역방향 별도 패스

## 와이어프레임 형식

각 화면은 ASCII 박스로 표현합니다. 각 와이어프레임 아래에 반드시:
- 요소 설명, 인터랙션 노트, BR 매핑을 포함하세요.

## 품질 자가 점검

- [ ] 기능 명세 화면 커버리지 = 100%
- [ ] 상태 매트릭스 빈 셀 = 0
- [ ] 모든 화면에 뒤로가기/이탈 경로 정의됨
- [ ] 디자인 요소 포함 = 0건
